SIP4AI Manager - Current Situation and Required Changes

The current application is working as a CRUD manager, but it is not functioning as a SIP4AI deployment platform.

What currently works

The application successfully:

* manages Clients
* manages Extensions
* manages Agent Configurations
* stores everything in PostgreSQL
* generates config.json
* generates the systemd service file
* serves the frontend correctly
* exposes the REST API

The dashboard is operational.

The problem

At the moment, after creating a client, extension and agent configuration, the application only lets me download:

* config.json
* systemd service

This is basically a configuration generator.

That is not what I wanted.

⸻

What SIP4AI actually does

The original SIP4AI installation is extremely simple.

curl -sSL https://sip4ai.com/install.sh | bash
export SIP_USERNAME=1004
export SIP_PASSWORD=xxxx
export OPENAI_API_KEY=...
sip4ai

SIP4AI immediately:

* registers the SIP extension
* authenticates to the PBX (Yeastar, FreeSWITCH, etc.)
* answers calls
* connects to the AI provider
* bridges RTP audio

There is no manual config download after installation.

The running process itself handles everything.
