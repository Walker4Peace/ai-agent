The deployment architecture currently assumes every SIP4AI instance binds to UDP port 5060. This prevents running multiple deployed agents on the same server because the second instance fails with listen udp :5060: bind: address already in use.

The deployment system should treat each extension as an independent SIP client process with its own allocated resources.

Required changes:

* Make the local SIP listening port configurable (e.g. SIP_LOCAL_PORT).
* Automatically allocate a unique SIP port for every deployment (5060, 5062, 5064… or any available port).
* Keep the Yeastar server on 5060; only the local client port changes.
* Allocate a unique HTTP API port per instance.
* Store sip_local_port, http_port, pid, service_name, and status in the database.
* Generate and launch each instance as its own systemd service (e.g. sip4ai-1001.service, sip4ai-1002.service).
* Ensure the SIP REGISTER Contact header advertises the correct local listening port for each instance.

The goal is to support dozens or hundreds of simultaneously registered SIP extensions on a single server, similar to how Zoiper or Linphone can run multiple SIP accounts. The deployment manager should automatically handle port allocation and process management so users only deploy an extension without worrying about networking.
